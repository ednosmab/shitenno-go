# Principles

> Immutable principles that govern every decision in the Nexus System.

These principles cannot be overridden by convenience, by technology choices, or by market pressure. They are the identity of the Nexus. Every architecture decision, every domain model, every implementation must be consistent with them.

---

## 1. Code is Consequence of Knowledge

Code exists because knowledge was organized. Knowledge does not exist because code was written.

When a team understands its domain, its patterns, and its constraints, the code follows naturally. When code is written without understanding, it accumulates debt that compounds over time.

Nexus organizes the knowledge first. The code is the result.

---

## 2. Architecture is Consequence of Domain

Architecture does not exist for its own sake. It exists to serve the domain.

The domain defines what matters. The architecture defines how to organize it. When architecture diverges from domain, it becomes overhead. When architecture serves domain, it becomes leverage.

Nexus models the domain of engineering governance. The architecture is a consequence of that model.

---

## 3. Capabilities Evolve Before Features

A capability is a unit of governance maturity. A feature is a unit of functionality.

Features can be added without understanding. Capabilities cannot. When a team installs a capability, it is committing to a new dimension of governance. This commitment must be earned through knowledge, not assumed through installation.

Nexus recommends capabilities based on maturity. It never forces installation based on labels.

---

## 4. AI Amplifies Good Engineering

AI does not fix bad engineering. It amplifies whatever exists.

A project with organized knowledge, clear decisions, and governed context becomes more powerful with AI. A project without these things becomes more chaotic with AI.

Nexus provides the structure that makes AI useful. It ensures that AI agents receive governed, hierarchical context. It ensures that AI recommendations are grounded in evidence, not speculation.

---

## 5. Every Decision Generates Knowledge

A decision that is not recorded is a decision that will be repeated.

Every decision made in a project — architectural, technical, procedural — generates knowledge. That knowledge must be captured, connected, and made available for future decisions. Otherwise, the team is condemned to repeat itself.

Nexus tracks decisions through the knowledge lifecycle. It ensures that knowledge compounds instead of decaying.

---

## 6. Engineering State is More Important Than Code State

The state of the code tells you what exists. The Engineering State tells you what it means.

A project can have clean code and broken governance. It can have high test coverage and zero architectural records. It can have active development and decaying knowledge.

The Engineering State measures what matters: the maturity of governance, the health of knowledge, the readiness of capabilities. It is the source of truth for understanding a project's actual condition.

Nexus makes the Engineering State visible, measurable, and actionable.

---

## How These Principles Interact

```
Knowledge generates code (Principle 1)
    ↓
Code serves domain (Principle 2)
    ↓
Domain evolves through capabilities (Principle 3)
    ↓
AI amplifies the result (Principle 4)
    ↓
Every step generates knowledge (Principle 5)
    ↓
The cycle is measured by Engineering State (Principle 6)
```

These six principles form a closed loop. They reinforce each other. Breaking one weakens all.

---

*These principles are permanent. They define the identity of the Nexus System. They cannot be changed without changing what Nexus is.*
